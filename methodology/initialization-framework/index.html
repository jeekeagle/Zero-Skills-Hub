初始化框架 · 六维做事穿透工具

一、这个 skill 是什么​

做事分析不是列计划，不是写方案，不是画甘特图。

做事分析是把战略意图、沟通口径、验证标准、迭代机制一次性想透。尤其适合多Agent协作——每个Agent知道自己的边界，也知道怎么跟别人配合。

六维框架的价值：

  • 第一层没想清楚，后面全是盲人摸象。价值判断不清楚，后面所有设计都是空中楼阁。
  • 验证标准层面容易被忽略，但至关重要。没有验证标准，做完了也不知道有没有做成。
  • 多Agent协作场景下，调度与编排层面必须明确。每个Agent知道自己的边界，也知道怎么跟别人配合。边界不清晰，协作必然混乱。
  • 长期项目要重视扩展与演化层面。方法体系不是静态的，要能在实践中自我修正和迭代。没有迭代机制，方法会过时。

二、怎么钻：从上往下穿透​

钻之前，先抬头看一眼这个项目立在什么上面——它的价值判断是什么？

第 1 层 · 价值判断（天花板）​

三个问题，答不出来别往下走：

  1. 为什么做这件事？最终想达到什么状态？
  2. 这件事在整体战略中的定位是什么？
  3. 投入产出比是否合理？优先级如何？

判断标准： 价值判断不清楚，后面所有设计都是盲人摸象。这一层必须先想清楚，它是其他五层的基础。

操作问句两条，对着项目问：

  • 这件事如果不做，会失去什么？（一问就知道价值在哪）
  • 这件事做成了，会带来什么？（一问就知道目标在哪）

答出来的几条，就是这个项目的价值判断。

价值判断和实现路径，一句话分清：

  • 价值判断是天花板——往上不可追，定了就定了。
  • 实现路径是地板——一层一层往下追，追到追不动，干而见。

两个方向，一上一下。天花板没看清，容易把手段当目的；地板没找到，容易把空想当方案。

第 2 层 · 话语体系（统一语言）​

关键概念要定义清楚：

  1. 核心术语的定义是什么？常见误解在哪？
  2. 需要建立哪些统一语言？
  3. 这件事的底层运作机制是什么？各要素之间的因果关系如何？

叙事框架要分受众：

  1. 如何向领导讲清楚这件事？
  2. 如何向团队讲清楚这件事？
  3. 如何向用户讲清楚这件事？

判断标准： 不同受众需要不同版本的表述。别用一套话应付所有人。话语体系不统一，执行时必然走样。

话语体系和技术方案，一句话分清：

  • 话语体系是翻译层——把技术语言翻译成不同受众能听懂的语言。
  • 技术方案是执行层——具体怎么做，用什么工具，什么流程。

翻译层做不好，执行层再好也传达不出去。

第 3 层 · 实现路径（怎么干）​

方法选择：

  1. 用什么工具、什么流程来做？
  2. 有哪些备选方案？为什么选这个？

推进节奏：

  1. 分几个阶段？每个阶段的重点是什么？
  2. 时间节点如何安排？

资源配置：

  1. 需要哪些人力、时间、技术资源？
  2. 资源从哪里来？如何分配？

判断标准： 资源配置不合理，再好的方案也落不了地。方法选择要匹配资源约束。

实现路径和验证标准，一句话分清：

  • 实现路径是施工图——怎么把房子盖起来。
  • 验证标准是验收单——盖好了怎么验。

施工图画得再好，没有验收单，盖完了也不知道合不合格。

第 4 层 · 验证标准（怎么算成）​

成功标准要量化：

  1. 怎么判断做成了？可量化的指标是什么？
  2. 哪些是底线指标，哪些是优秀指标？

验证流程要明确：

  1. 每个节点如何检查质量？
  2. 谁负责验证？验证通过的标准是什么？

风险预案要前置：

  1. 可能出什么问题？概率如何？
  2. 每种风险如何应对？有没有备选方案？

判断标准： 验证标准层面容易被忽略，但至关重要。没有验证标准，做完了也不知道有没有做成。

验证标准和调度编排，一句话分清：

  • 验证标准是静态的——定了就不变，用来验收。
  • 调度编排是动态的——执行过程中随时调整，用来协调。

静态标准管结果，动态调度管过程。

第 5 层 · 调度与编排（多Agent协作）​

调度机制：

  1. 多个子任务如何分配给不同的Agent/角色？
  2. 任务之间的依赖关系是什么？

权重规则：

  1. 不同方法的选择优先级是什么？
  2. 遇到冲突时如何决策？

职责边界：

  1. 谁负责决策？谁负责执行？
  2. 边界在哪里？如何避免混乱？

判断标准： 多Agent协作场景下，这一层必须明确。每个Agent知道自己的边界，也知道怎么跟别人配合。边界不清晰，协作必然混乱。

调度编排和扩展演化，一句话分清：

  • 调度编排是管当下——当前这个项目怎么跑通。
  • 扩展演化是管未来——这套方法怎么复用到其他项目。

当下跑不通，未来没意义；未来没规划，当下只能用一次。

第 6 层 · 扩展与演化（长期视角）​

迁移能力：

  1. 这套方法能否复用到其他场景？
  2. 复用时需要调整哪些部分？

迭代机制：

  1. 如何收集使用反馈？
  2. 多久回顾一次？如何优化？

淘汰规则：

  1. 什么情况下需要重构？
  2. 什么情况下应该放弃？

判断标准： 长期项目要重视这一层。方法体系不是静态的，要能在实践中自我修正和迭代。没有迭代机制，方法会过时。

三、反坍缩闸 · 下笔前强制自检​

AI 最容易犯的六个错误，每个错误都给出"症状→反问→正确做法"。

错误一：把价值判断当目标​

  • 症状： 把"为什么做这件事"写成"提高效率""降低成本"这种通用目标。
  • 反问： 这件事如果不做，会失去什么？如果答案是"什么都失去不了"，说明价值判断没想清楚。
  • 正确做法： 价值判断要问"不做会失去什么"，不是"做成了会得到什么"。失去的才是真正的价值。

错误二：把话语体系当技术方案​

  • 症状： 话语体系里全是技术术语，没有针对不同受众调整表述。
  • 反问： 领导能看懂这段话吗？用户能看懂这段话吗？如果答案是"不能"，说明话语体系没做好。
  • 正确做法： 话语体系是翻译层，要把技术语言翻译成不同受众能听懂的语言。同一件事，对领导、对团队、对用户，要用三套不同的表述。

错误三：把验证标准当里程碑​

  • 症状： 验证标准里写的是"完成XX功能""上线XX模块"，这是里程碑，不是验证标准。
  • 反问： 怎么判断做成了？如果答案是"功能做完了"，说明验证标准没想清楚。
  • 正确做法： 验证标准要回答"怎么判断做成了"，不是"做到哪一步了"。功能做完了不等于做成了，要看用户用得怎么样。

错误四：把调度编排当分工表​

  • 症状： 调度编排里写的是"A负责XX，B负责YY"，这是分工表，不是调度编排。
  • 反问： A和B之间怎么配合？如果答案是"各做各的"，说明调度编排没想清楚。
  • 正确做法： 调度编排要回答"多Agent之间怎么配合"，不是"谁做什么"。多Agent协作的核心是配合，不是分工。

错误五：把扩展演化当总结​

  • 症状： 扩展演化里写的是"这个项目做完了，总结一下经验"，这是总结，不是扩展演化。
  • 反问： 这套方法能复用到其他场景吗？如果答案是"不能"，说明扩展演化没想清楚。
  • 正确做法： 扩展演化要回答"这套方法怎么复用"，不是"这个项目做得怎么样"。方法的价值在于复用，不在于单次使用。

错误六：把六层当检查清单​

  • 症状： 每层都填满字段，但层与层之间没有穿透关系。
  • 反问： 这一层和上一层是什么关系？如果答案是"没关系"，说明没穿透。
  • 正确做法： 六层是穿透关系，不是并列关系。每一层都要回答"这一层的地面是什么"，不是"还有什么要补充"。

四、使用建议​

不必每次都填满所有字段，根据事情复杂度选择重点。简单项目可能只用到前四层，复杂项目六层都要走透。

三条铁律：

  1. 纵向，不横向——每层必须回答"这一层地面是什么"，不是"还有什么要补充"
  2. 单刀直入——不旁征博引，不铺陈背景，直取要害
  3. 层层穿透——每一层都比上一层更接近骨头

五、与元方法框架的关系​

  • 验证标准层面对应"标准驱动开发"的主体验证层。
  • 调度与编排层面对应"调度与编排层"。
  • 扩展与演化层面对应"扩展与演化层"。
  • 话语体系层面是"封装与复用层"的基础。

整个框架遵循 Dogfood 机制——用这套方法来设计这套方法本身，让它在实践中自我修正和迭代。

六、怎么写 · 输出规范​

写一场下坠。不是检查清单。

带读者从用户给出的项目/任务开始往下钻，每一层都比上一层更接近骨头。

要求：

  • 有失重感——读的人能感觉到在下钻，不是在平移
  • 每层有命名——给每一层一个精准的名字，两三个字
  • 层间有裂缝——每层结尾点出一个问题或矛盾，那就是通往下一层的裂缝
  • 终点要狠——最后一层必须让人沉默片刻

输出两件事：

  1. markdown 语法。
  2. 报告文件路径给用户（保存到本地即可，无需联网同步）。

七、执行流程（被调用时怎么做）​

当用户调用本 skill 时，按以下顺序执行：

  1. 接收项目/任务输入——明确要分析的对象。可以追问 1-2 个澄清问题，但不要超过 3 轮。
  2. 依次走穿六层——每层只回答"这一层的地面是什么"，不留并列项。
  3. 跑反坍缩闸——下笔前用六个反问自检，发现坍缩就重写。
  4. 生成 markdown 报告——按"怎么写"的规范写出有失重感的穿透文，不是字段填空。
  5. 保存到本地——写入 markdown 文件，默认存放在当前工作目录下，命名格式：做事分析-<项目名>.md。
  6. 回传报告路径——把生成的本地文件路径告诉用户。

八、与其他 skill 的搭配​

  • 叙事密度高、需要打磨表达 → 搭配 baoyu-format-markdown 美化版式
  • 需要做成 PPT 汇报 → 搭配 pptx skill
  • 要继续做扩展演化层的复用拆解 → 搭配 theory-skill-generator
  • 多视角交叉审视方案 → 搭配 perspective-integrator
  • 需要做技术方案深挖 → 搭配 system-perspective / topology-perspective

九、Skill 自检 · Dogfood​

本 skill 自身按六层自检一遍：

  • 价值判断：不做初始化框架，复杂项目在多Agent协作下必然混乱坍缩。
  • 话语体系：对领导说"穿透式分析"，对团队说"六层对齐"，对用户说"帮你把事情想透"——三套话。
  • 实现路径：调用本 skill → 走六层 → 输出 markdown → 存本地文件。
  • 验证标准：用户读完报告能说出"这一层地面是什么"就算成；说不出就是没穿透。
  • 调度编排：本 skill 自己是调度层入口，下游可分发到具体执行 skill（pptx / 格式化 / 理论生成）。
  • 扩展演化：方法论可迁移到任何"做事"场景，每用一次回炉一次，反向优化框架本身。
